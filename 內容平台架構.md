# SonaSNS 內容平台架構 — 品牌頁 / BLOG / 行業資訊 / 生成器 串接

> 一句話:**一條內容管線餵所有下游。** 排程 agent 研究+寫作 → 各品牌內容資料夾 →
> 品牌頁、生成器、每日 Sona 草稿全部從同一份資料讀。BLOG 是脊椎,同時是素材庫。
> 最後更新:2026-07-18(規劃;SS 有 Token 時分階段蓋)

## 全景:一條龍怎麼串

```
① 題材來源            ② 排程 agent(雙週/品牌)         ③ 內容資料夾            ④ 下游消費
──────────           ─────────────────────           ──────────            ──────────
weekly_topics.json   研究(上網查證·必附來源)          content/<brand>/       品牌頁 3 tab:
下週題材建議.md   →   依 Brand DNA 寫:            →    news.json         →   概覽/行業資訊/BLOG
(各品牌各一份)         · 1 篇 BLOG 常青文               blog/*.md +index      生成器(tools.html):
                      · 2–4 則行業資訊(附源)          social_pool.json      · 最新=讀 social_pool
                      · 回填 social_pool(Threads+IG)                        · 固定題材=抓 BLOG 重點
                                                                            每日 Sona 草稿:引用素材
```

**關鍵洞察**:BLOG / 行業資訊 / 生成器不是三件事,是同一條管線的三個出口。管線建一次,三個一起亮。

## 各品牌落點(重要:不是都在同一個 repo)

| 品牌 | 品牌頁位置 | 行業資訊 + BLOG 長在哪 |
|------|-----------|----------------------|
| OKIBLUES | SonaSNS `okiblues.html` | **就在 SonaSNS**(content/okiblues/) |
| OKIPLAYGROUND | SonaSNS `okiplayground.html` | **就在 SonaSNS**(content/okiplayground/) |
| DiveInOut | SonaSNS `diveinout.html`(概覽+帖文+廣告成效) | **在 SakiDive repo**;Sona 頁的兩個 tab 直接**連出去**到 SakiDive 的 news.html / blog.html |
| THE BLUES | SonaSNS `theblues.html` | **不做** 行業/BLOG(跟 DiveInOut 潛水內容重疊);只做品牌資訊 + YouTube成效 + Google評價 + 廣告報告 |

> 格式統一:SakiDive 的 `BLOG架構.md` 的 news.json schema + 時間軸列表 + 必附來源,是
> OKIBLUES/OKIPLAYGROUND 行業資訊頁的**參考格式**,只是換行業(沖繩租車 / 沖繩旅遊)。

## 品牌頁結構(OKIBLUES / OKIPLAYGROUND)

概覽上方加分頁列(tab bar):`📋 概覽 ｜ 📰 行業最新資訊 ｜ ✍️ BLOG`
- **概覽**:現有內容(品牌資料/賣點/YouTube/評價/廣告報告)
- **行業最新資訊**:client-side 讀 `content/<brand>/news.json`,倒序時間軸,每則附來源
- **BLOG**:讀 `content/<brand>/blog/index.json` 列表 → 單篇 md 用內嵌轉譯器渲染。
  受眾導向寫(OKIBLUES→要來沖繩租車的客戶;OKIPLAYGROUND→計劃來沖繩玩的華人)

## 生成器(tools.html)升級

現況:10 產業人設 + 固定範本,只出 Threads。要改成:

1. **人設區加三個 Brand**(OKIBLUES / OKIPLAYGROUND / DiveInOut),沿用各自 DNA(`brands/*.md`)
2. **新增 IG 帖文輸出**(現在只有 Threads):同一主題產 Threads 版 + IG 版兩塊,各自可複製
3. **兩種來源**:
   - **🆕 最新**:讀該品牌 `social_pool.json`(排程 agent 依當週新文章/行業資訊 + DNA 寫的新鮮草稿)
   - **📌 固定題材**:從該品牌**已發佈 BLOG / DNA 的公司常青重點**抓來套版(如 OKIBLUES 車隊/賣點、某篇攻略要點)——SS 要的「從 Blog 公司相關資料抓出來寫」
4. 原本 10 個泛產業人設保留(給沒有 Brand 的通用情境)

## 排程(全部沿用今天建好的 claude-code-action + cron 那套)

| 排程 | 頻率 | 產出 |
|------|------|------|
| Saki(潛水) | **雙週**(原每月,SS 要密→建議 14 天) | SakiDive:1 BLOG + news + diveinout social_pool |
| OKIBLUES agent | 雙週 | SonaSNS:1 BLOG + news + social_pool |
| OKIPLAYGROUND agent | 雙週 | SonaSNS:1 BLOG + news + social_pool |
| (現有)每日 Sona 草稿 | 每日 | Sheet 帖文草稿(可引用 BLOG 素材) |

鐵則(所有 agent):資訊上網查證、**必附來源**、缺料寫【待補】不編造;只寫內容不亂動別的檔。

## 建置階段(先做案例再抽象)

1. **Phase 1 — OKIBLUES 一條龍當範本**:品牌頁 3 tab + content/okiblues/ 資料模型 + 種 1 篇 BLOG&幾則資訊 + 生成器 OKIBLUES 模式(含 IG 輸出) → SS 確認滿意
2. **Phase 2 — 複製到 OKIPLAYGROUND**
3. **Phase 3 — 排程 agent**(OKIBLUES/OKIPLAYGROUND 雙週;同時把 Saki 改雙週 + 升級寫 BLOG 結構)
4. **Phase 4 — DiveInOut**:SakiDive 蓋 news/blog 兩頁 + Sona diveinout.html(概覽+帖文+成效,兩 tab 連出去)
5. **Phase 5 — THE BLUES 基本頁**(無行業/BLOG)
6. 完善後再展其他品牌
